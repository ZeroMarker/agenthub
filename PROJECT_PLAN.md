# AgentHub 项目计划

> 版本：v1.0  
> 更新日期：2026-06-14  
> 当前阶段：可演示原型（Prototype）

## 1. 项目目标

AgentHub 面向需要同时使用多个 AI 编程助手的开发者，提供统一的代理目录、安装状态检测、安装/卸载和信息查询能力，并同时支持命令行与桌面界面。

v1.0 的核心目标：

- 使用一份可信的数据源维护所有代理信息，CLI 与 GUI 行为保持一致。
- 在 Windows、macOS 和 Linux 上执行明确、可审计的安装与卸载流程。
- 准确展示代理是否已安装、已安装版本和可用安装方式。
- 安装失败时提供可操作的错误信息，不让部分失败被误报为成功。
- 建立自动化测试、持续集成和可重复发布流程。

## 2. 当前基线

截至 2026-06-14，项目已具备：

- Rust CLI：支持 `list`、`search`、`info`、单个/批量安装和卸载。
- Tauri 2 + Vue 3 桌面原型：支持搜索、筛选、排序、网格/表格视图和批量操作。
- 40 个候选代理的静态目录。
- Windows PowerShell 构建与冒烟测试脚本。

当前实现仍属于原型，主要差距如下：

| 优先级 | 差距 | 影响 |
|---|---|---|
| P0 | CLI 与 Tauri 后端各维护一份代理数据和安装逻辑 | 数据容易漂移，修复需要重复修改 |
| P0 | 多数桌面代理被标记为 `npm` 管理器 | 安装、卸载和状态检测可能执行错误命令 |
| P0 | 代理名称、包名、winget ID、brew cask 混用 | 跨平台行为不可靠 |
| P0 | 安装命令直接执行，缺少预览、确认和输入约束 | 存在误安装、误卸载和供应链风险 |
| P1 | 状态检测依赖文本包含匹配，且命令可能较慢 | 可能误判，列表加载时间不可控 |
| P1 | 缺少 Rust 单元/集成测试和前端组件测试 | 回归风险高 |
| P1 | GUI 安装操作阻塞后端线程，批量操作缺少逐项进度 | 用户体验和错误恢复较弱 |
| P1 | 文档中的包版本和“已验证”状态是静态信息 | 信息会快速过期 |
| P2 | 缺少 CI、发布产物、签名、更新机制和贡献指南 | 无法稳定交付给外部用户 |

## 3. v1.0 范围

### 必须完成

- 共享代理清单与统一 Rust 核心库。
- npm、pip、winget、brew 四类安装器适配。
- CLI 和 GUI 的列表、搜索、详情、状态检测、安装及卸载。
- 安装命令预览、交互确认、`--yes` 非交互选项和 `--dry-run`。
- 单项与批量操作结果明细、超时和错误分类。
- Windows、macOS、Linux 的自动化测试与构建。
- 版本化发布说明和可复现的安装文档。

### 暂不纳入 v1.0

- 代理账号登录、API Key 托管和云端同步。
- 自动配置各代理的模型、提示词或 MCP 服务。
- 第三方插件市场。
- 遥测、推荐算法和团队管理。
- 对所有目录代理承诺“一键安装”；无可靠包管理器来源的代理仅打开官网。

## 4. 目标架构

```text
agents.json / agents.toml
        |
        v
agenthub-core (Rust)
  - catalog
  - platform detection
  - installer adapters
  - status/version detection
  - operation results
     |                 |
     v                 v
agenthub-cli       Tauri commands
                         |
                         v
                      Vue UI
```

关键设计原则：

- **单一事实来源**：代理元数据只维护一次，并通过 schema 校验。
- **安装器适配层**：不同包管理器负责生成命令、检测状态和解析版本。
- **平台明确化**：每个代理按平台声明安装方式，不使用模糊的通用 `manager` 字段。
- **默认可审计**：执行前可查看命令；日志隐藏敏感信息并记录结果。
- **可测试性优先**：命令执行器可替换为 mock，测试不实际安装软件。

建议的代理清单字段：

```yaml
id: cursor
name: Cursor
kind: desktop
provider: Cursor
homepage: https://cursor.com
installers:
  windows:
    manager: winget
    package: Anysphere.Cursor
  macos:
    manager: brew-cask
    package: cursor
  linux:
    manager: manual
```

## 7. 验收标准

v1.0 发布前必须满足：

- CLI 与 GUI 对同一清单返回一致的代理数量和元数据。
- 所有 `verified` 安装器均在声明的平台完成安装、检测和卸载测试。
- 无可靠安装方式的代理只提供官网入口，不执行猜测命令。
- `--dry-run` 不修改系统，并输出将执行的完整操作。
- 非交互安装必须显式传入 `--yes`；GUI 必须有确认步骤。
- 批量操作中单项失败不影响结果汇总，且退出状态准确。
- 核心库关键模块测试覆盖率目标不低于 80%。
- CI 在 Windows、macOS、Linux 上通过格式、lint、测试和构建。
- README、支持矩阵和发布说明与当前版本一致。

## 8. 测试策略

| 层级 | 内容 | 工具/方式 |
|---|---|---|
| 单元测试 | 清单解析、筛选、命令生成、版本和状态解析 | Rust `cargo test` |
| 契约测试 | 每种安装器的参数、输出解析与错误映射 | Mock command runner + fixtures |
| 前端测试 | 搜索、筛选、选择、进度和错误展示 | Vitest + Vue Test Utils |
| 端到端测试 | CLI 常用命令、Tauri 关键流程 | 临时环境或受控测试包 |
| 平台冒烟 | 构建、启动、dry-run、手动安装入口 | GitHub Actions matrix |
| 发布回归 | 安装、升级、卸载、校验和 | 每个发布候选版本 |

自动化测试不得在普通 CI 环境安装真实的第三方 AI 工具；真实安装验证应在隔离的发布测试环境执行。

## 9. 风险与应对

| 风险 | 概率/影响 | 应对 |
|---|---|---|
| 第三方包名、安装方式或产品状态频繁变化 | 高/高 | 为清单记录验证日期，定期检查，允许标记 deprecated |
| winget、brew、npm、pip 输出格式变化 | 中/高 | 使用结构化输出，保留解析 fixture 和契约测试 |
| 安装命令拥有较高系统权限 | 中/高 | 默认确认、命令预览、最小权限、来源展示 |
| 部分代理没有可靠的包管理器安装方式 | 高/中 | 降级为 manual，只打开官网，不猜测命令 |
| 跨平台测试资源有限 | 中/高 | CI matrix + 发布前隔离环境人工验证 |
| 目录规模增长导致维护成本上升 | 中/中 | schema、生成文档、贡献模板和自动校验 |

## 10. 成功指标

- 目录元数据重复维护点：从 2 处降为 1 处。

- CLI 列表命令在常规环境中的 P95 响应时间：小于 3 秒。
- 安装失败的可诊断率：100% 返回明确的代理、命令阶段和错误摘要。
- 发布后首月严重安装误报：0 个。
- 文档支持矩阵与清单自动一致率：100%。
